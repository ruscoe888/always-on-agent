How To Use This In Your Daily Workflow
Reading an agent-filed issue

Every issue the agent creates follows the same format. Here's what each section means and what you need to do with it.

TITLE
[Agent Triage] tells you the agent filed it. The severity badge (P1) is in the title so you can triage the list without opening each issue.
SEVERITY
The agent's assigned severity with its reasoning. Treat this as an informed first opinion — you can override it. If you disagree, update the issue and document why so we can improve the rubric.
RUNBOOK
The matched runbook file. If matched: the recommended fix steps are in the body — act on them. If "No matching runbook found": the incident is novel; you'll need to diagnose manually and should consider adding a new runbook.
ESCALATION
Which on-call team to page. Check this first for P0/P1 — the agent will have identified the right team based on the service area.
Important: The agent files issues — it does not close them. Once you've resolved an incident or addressed a compliance violation, close the GitHub Issue manually. This keeps the dashboard accurate.

CLOSING AN ISSUE
When the incident is resolved Close the GitHub Issue manually. Leave a comment with the resolution. The agent will not re-open it.
When the compliance violation is remediated Close the compliance issue and note the contract version that was updated. Legal team should confirm.
When the severity is wrong Edit the title, leave a comment. The agent reads its own PROD-ID to dedup — your title edit won't be overwritten.
WHAT THE AGENT DOES NOT DO
✗Page on-call directly (coming next — needs webhook)
✗Close resolved issues — you do that
✗Diagnose issues without a matching runbook — it flags them as novel
✗Execute rollbacks — it recommends, you action
✗Read live monitoring systems — it reads the repo only
Making It Better — Your Input Matters
The agent learns from your feedback

The agent's behaviour is defined in CLAUDE.md — a plain text file committed to the repo. Every time the ops team sees a wrong severity, a missed correlation, or a runbook that's out of date, that's a signal we can act on.

📖

Wrong runbook match?

Tell us the PROD-ID and which runbook should have matched. We'll update the keyword table in CLAUDE.md.

🔗

Missed a correlation?

If the agent didn't link an incident to a deploy that you knew was the cause, let us know the PROD-ID and deploy. We'll check if the timestamp window needs widening or the service-area matching needs updating.

⚖️

New compliance rule?

If legal adds a new policy requirement, it gets appended to compliance-policy.md. The agent picks it up on the next run automatically — no code change needed.

📌

New runbook?

Add a markdown file to runbooks/ and add keyword entries to the Task 1 section of CLAUDE.md. The agent will match against it from the next run.
