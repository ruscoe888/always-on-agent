Architecture
Two-cron design: ops-agent + fabricator, both backed by CLAUDE.md

Two durable crons are live. The ops-agent cron fires every 4h at :17 — it does triage, correlation, and compliance. The fabricator cron fires every 20 minutes — it runs scripts/fabricate-issue.py to synthesise realistic PROD-XXXX JSON files so the ops-agent always has fresh data to process. Both crons keep their prompts minimal and delegate all domain logic to CLAUDE.md.

ops-agent cron (every 4h :17)
~150-word prompt. Tells Claude to git pull and run the full ops protocol from CLAUDE.md — triage, correlation, compliance, dedup, summary.
|
fabricator cron (every 20 min)
Invokes scripts/fabricate-issue.py. Picks one of 6 templates, auto-increments PROD-ID with a random 1–8 gap, commits and pushes. Dev-only — supplies test data for the ops-agent.
↓ ops-agent flow
git -C /repo pull origin main
Fetches latest data including fabricator-generated issues. Repo is single source of truth.
→
CLAUDE.md loaded
Auto-loaded by Claude Code. Contains: severity rubric, runbook keywords, correlation logic, compliance rules, dedup query, output format.
→
gh issue list (dedup)
Queries existing ops-agent issues before creating. Matches by PROD-XXXX ID or contract+rule string. Skips dupes.
→
gh issue create (×N)
One issue per net-new finding. Labels: ops-agent + task-specific. Structured body with source file refs.
Why not a single monolithic prompt? — Cron prompts are string literals. A 2000-word prompt is fragile, unversioned, and can't be reviewed in a PR. CLAUDE.md is versioned in git, diffable, and Claude reads it natively. Changing agent behaviour = commit to CLAUDE.md = reviewed change. Docs live in docs/ (HLD.html, LLD.html, plan-2-architecture.md, agent-roster.md, Plan.md).
Permissions Model
Why the agent ran without prompting — 11 allowlisted commands

Claude Code prompts for any Bash command not in an explicit allowlist. In an unattended cron run, a prompt = a hang. We solved this at two layers:

GLOBAL
~/.claude/settings.json — 11 commands: git pull, git -C * pull, gh issue list/create, gh label list/create, gh api, cat, ls, find, echo. Applies to all Claude Code sessions including both crons.
PROJECT
.claude/settings.json (committed to repo) — mirrors the same allowlist. Provides a fallback and documents what the agent needs if run on a different machine.
NOTE
The fabricator cron invokes scripts/fabricate-issue.py via python3 — but python3 is called directly in the cron prompt, not via the Bash tool. The allowlist therefore does not need a Bash(python3 *) entry.
GAP
Any novel Bash pattern not in the allowlist will still prompt and hang the cron. If you add new shell commands to CLAUDE.md, add them to the allowlist first.
Security note: The allowlist uses prefix matching — Bash(gh issue create*) allows any gh issue create invocation. If the agent ever reads untrusted data that could influence its tool calls, this is a prompt injection surface. The current data (JSON/markdown in a controlled repo) is low-risk.
~/.claude/settings.json — 11 commands
// Global — applies to all sessions, including both crons
{
  "permissions": {
    "allow": [
      "Bash(git pull*)",
      "Bash(git -C * pull*)",
      "Bash(gh issue list*)",
      "Bash(gh issue create*)",
      "Bash(gh label list*)",
      "Bash(gh label create*)",
      "Bash(gh api*)",
      "Bash(cat *)",
      "Bash(ls *)",
      "Bash(find *)",
      "Bash(echo *)"
    ]                    // 11 total
  }
}

The Agent Contract
CLAUDE.md is the source of truth

CLAUDE.md defines the agent's behaviour. It is loaded automatically by Claude Code before any run. Think of it as a system prompt that lives in version control.

SECTION 1

Ops Triage Protocol

Severity rubric (P0–P3), runbook keyword matching, on-call routing. Deterministic — same input always produces the same severity label.

SECTION 2

Incident Correlation

48-hour window, service-area matching, causal chain detection. Relies on ISO 8601 timestamps in issues/ and deploys/recent.json.

SECTION 3

Compliance Audit

7-rule policy check per contract. References compliance-policy.md at runtime — policy changes in that file are reflected immediately on the next run.

SECTION 4

Deduplication + Output

Issue title format, label schema, dedup query logic, and run summary format. Deterministic enough to be tested manually.

Extending the agent: Add a new runbook? Update the keyword table in Section 1. New compliance rule? Append to compliance-policy.md and add the check to Section 3. New output channel (Slack, PagerDuty)? Add a new section to CLAUDE.md with the relevant Bash commands — then allowlist them. Plan 2 note: the agents/ directory will house per-agent CLAUDE.md files for the six specialised Plan 2 agents (Watcher, Triage, Correlation, Compliance, Fabricator, Keeper).
CLAUDE.md — severity rubric excerpt
## Task 1: Ops Triage

Assign severity using this rubric:
- P0: Multiple customers affected,
       revenue impact, active outage
- P1: Single enterprise customer affected,
       or service degraded
- P2: Intermittent, workaround available,
       limited user impact
- P3: Feature requests, non-urgent

Runbook keyword matching:
- auth-502-windows.md
  → 502, login, auth, connection pool
- cdn-upload-latency.md
  → upload, CDN, slow, latency, image
- payment-service-degraded.md
  → PaymentService, checkout, NPE

Data Layer
What the agent reads — and what it expects
issues/PROD-XXXX.json
{
  "id": "PROD-4521",
  "title": "...",
  "status": "open",
  "severity": null,   ← agent fills
  "labels": [],
  "assignee": null, ← agent fills
  "opened_at": "ISO8601",
  "reporter": "email",
  "body": "...",
  "comments": []
}
Critical field: opened_at — used for the 48h correlation window. Must be ISO 8601. Missing or malformed timestamps will silently break correlation. The agent-fabricated label is applied by the fabricator to all synthetic issues so the ops-agent can distinguish them if needed.
deploys/recent.json
{
  "deploys": [{
    "service": "payment-service",
    "version": "v4.8.2",
    "deployed_at": "ISO8601",
    "deployed_by": "email",
    "commit": "9f4a1c8",
    "summary": "...",
    "files_changed": ["..."],
    "rollback_available": true,
    "last_known_good": "v4.8.1"
  }]
}
Still manually maintained. The natural next step is a CI/CD pipeline that appends to this file on every deploy. Until then, correlation quality depends on a human remembering to update it after each release.

The agent reads this file at runtime on every run. Updating the policy requires only a commit — no changes to CLAUDE.md or the cron.
Design Decisions — Honest Assessment
What we chose and what we gave up
Decision What we chose Alternative considered Why we chose this What we gave up
Runtime Claude Code durable cron GitHub Actions scheduled workflow Zero infra, zero YAML, runs in existing Claude Code session 7-day TTL — must renew. Requires Claude Code running on a machine.
Output GitHub Issues via gh CLI Commit markdown reports to repo Issues are actionable, assignable, and closeable — reports are static Issues accumulate indefinitely. No auto-close on resolution.
Domain logic location CLAUDE.md in repo Embedded in cron prompt Versioned, reviewable, auto-loaded by Claude Code Agent must cd into the correct directory to pick it up
Deduplication gh issue list title-match Local state file / database No external state — GitHub is the source of truth Linear scan — slow at >200 issues. PROD-ID matching is substring, not exact.
Deploy data Manual JSON file (deploys/recent.json) Live CI/CD API query No auth tokens needed in the agent. Simple to demo. Stale if deploys aren't manually appended after each release. Not production-ready as-is.
Issue fabrication Python script cron (fabricate-issue.py, every 20 min) Manual test data entry Hands-free — always-fresh PROD-XXXX files for the ops-agent to process. 6 templates covering realistic failure modes. Fabricated issues enter the triage pipeline; ops-agent will open GitHub Issues for synthetic problems unless explicitly filtered.
Known Gaps and Extension Points
What to build next
KNOWN GAPS
1
7-day cron TTL — both crons Both the ops-agent cron (4h :17) and the fabricator cron (20 min) auto-expire after 7 days. Currently requires manual re-creation of each. Fix: promote to GitHub Actions on a schedule, or add a self-renewal cron that recreates both before expiry.
2
Deploy data is manual deploys/recent.json is hand-maintained. Should be appended by CI/CD on every deploy — a post-deploy GitHub Action step writing a JSON entry is the natural hook.
3
Dedup degrades at scale gh issue list --limit 200 is fine now. Beyond 200 issues, matches will be missed and duplicates filed. Add pagination or a state file if volume grows.
4
No P0 escalation path Agent files a GitHub Issue for a P0. That's it. A P0 needs a Slack or PagerDuty call. Add a Bash command to CLAUDE.md + an allowlist entry for curl to a webhook.
5
Dashboard GitHub API rate limit Dashboard polls GitHub API unauthenticated — capped at 120 req/h. With two active crons both triggering refreshes, this headroom is thin. Fix: add an auth token to the dashboard fetch calls to raise the limit to 5000 req/h.
EXTENSION POINTS
A
New runbooks Add markdown to runbooks/. Add keyword entries to the Task 1 section of CLAUDE.md. No other changes needed.
B
New compliance rules Append to compliance-policy.md. Add the check to Task 3 in CLAUDE.md. The agent picks it up on the next run.
C
New output channels Add a Task 4 to CLAUDE.md with the channel logic (Slack webhook, PagerDuty API, Jira ticket). Allowlist the new Bash commands globally.
D
Real-time trigger The 4h cron is a polling model. For sub-hour response, wire a GitHub webhook on issue creation to trigger the agent immediately via gh workflow dispatch or a Claude Code API call.
For Discussion
Open questions for the team
1
Who owns CLAUDE.md? It's a committed file — any engineer can PR a change. But unreviewed changes to the severity rubric or dedup logic could cause the agent to mis-triage or create duplicates. Should it require a designated reviewer?
2
Where should the cron live long-term? Currently tied to one machine running Claude Code. Plan 2 answers this concretely: GitHub Actions replaces Claude Code cron for production reliability and event-driven triggers (Watcher agent fires on push, <60s latency). Secrets move to ANTHROPIC_API_KEY (GitHub Secret) and FABRICATION_ENABLED (GitHub Variable). Each auth/cost profile is documented in plan-2-architecture.md.
3
How do we validate agent output quality? The agent's severity assignments and correlations are LLM outputs — plausible but not guaranteed correct. Should there be a human review gate before P0 issues page on-call? Or do we trust the rubric?
4
Is the prompt injection surface acceptable? The agent reads issue bodies from a controlled repo. If anyone can write to that repo, they can craft an issue body that attempts to manipulate the agent's tool calls. Access controls on the repo are the current mitigation.
5
What's the rollback story? If the agent files 50 incorrect issues due to a bad CLAUDE.md change, there's no bulk-close command. Should we add a cleanup script to the repo? Or is closing via gh issue list --label ops-agent sufficient?
6
Should fabricated issues be excluded from agent triage? Currently the ops-agent will triage agent-fabricated issues as if they were real — opening GitHub Issues for synthetic 503s, DB lag, TLS expiry etc. The agent-fabricated label exists but the ops-agent CLAUDE.md does not filter it. Should triage skip issues carrying that label, or is processing them a useful end-to-end smoke test?
