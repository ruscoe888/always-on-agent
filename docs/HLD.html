Section 01
Purpose & Scope

This document describes the high-level architecture of the Always-On Ops Agent: a system that continuously monitors a synthetic enterprise repository for production incidents, deployment correlations, and vendor compliance violations, then surfaces findings as structured GitHub Issues and a live operations dashboard.

It covers the deployed Plan 1 architecture (Claude Code durable cron), the designed Plan 2 architecture (GitHub Actions, specialised agents), the data model, the output model, and the implementation roadmap. For component-level detail, see the Low-Level Design.

Design philosophy: The agent is data-in, GitHub-Issues-out. The repository is the single source of truth. No external databases, no message queues, no persistent agent state beyond what lives in git and GitHub. Every component is replaceable independently.
Section 02
System Context

The system sits between a structured data repository and a human operations team. It has no user-facing API and no inbound traffic. It reads, reasons, and writes.

Section 03
Plan 1 — Current Deployed Architecture

Plan 1 is the live production system. A single Claude Code durable cron fires every 4 hours, reads the full repo state, runs three sequential analysis tasks, and files GitHub Issues for net-new findings. All agent logic lives in CLAUDE.md.

FIG 02 — Plan 1 Architecture. Single monolithic cron, sequential tasks, CLAUDE.md as the agent protocol.
COMPONENT

Durable Cron

Claude Code CronCreate. Fires 17 */4 * * *. Persisted to .claude/scheduled_tasks.json. 7-day TTL — must renew manually.

COMPONENT

CLAUDE.md

The agent brain. Loaded automatically when Claude Code enters the repo directory. Contains severity rubric, runbook keywords, correlation logic, compliance rules, dedup protocol, and output format.

COMPONENT

Fabricator Agent

scripts/fabricate-issue.py. 6 realistic issue templates. Auto-increments PROD-ID. Commits and pushes every 20 minutes. Dev/staging only.

COMPONENT

GitHub Issues

Output store. Labels: ops-agent, agent-triage, agent-correlation, agent-compliance, agent-fabricated. Dedup by title-matching.

COMPONENT

Dashboard

dashboard/index.html. Polls GitHub Issues API every 30s. 6 charts including a rolling 60-minute live activity feed. Click-to-drilldown on all charts.

COMPONENT

Repo (Data Store)

Single source of truth. issues/, deploys/recent.json, runbooks/, contracts/, compliance-policy.md. No database — git is the storage layer.

Section 04
Plan 2 — Target Architecture (GitHub Actions)

Plan 2 replaces the 4-hour polling cron with event-driven GitHub Actions. Six specialised agents replace the single monolith — each with a bounded responsibility, a small context window, and a specific trigger.

Key improvement: A new issue pushed to the repo triggers triage and correlation within 60 seconds, not up to 4 hours. The compliance agent runs daily. The keeper agent watches the agent ecosystem itself.
