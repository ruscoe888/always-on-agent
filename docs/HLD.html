Section 01
Executive Summary

Operations teams in modern enterprises are overwhelmed. The volume of production incidents, the complexity of distributed systems, and the breadth of vendor compliance obligations have outpaced the capacity of human teams to monitor, triage, and respond without introducing dangerous delays and coverage gaps — particularly outside business hours.

This document describes the high-level architecture of an Always-On Ops Intelligence Agent: an agentic AI system that continuously monitors the operational health of an enterprise environment, automatically triages every production incident, identifies root causes by correlating incidents with recent deployments, and audits vendor contracts for compliance violations — 24 hours a day, 7 days a week, without human initiation.

Conclusion first: This system has already filed 26 structured GitHub Issues across three domains — incident triage, deployment correlation, and vendor compliance — with zero permission prompts, zero human initiation, and response times of under 4 hours on current infrastructure, reducing to under 60 seconds in the target architecture.
Section 02
Problem Statement

The following problems were identified through discovery with the operations, engineering, and legal teams. They represent systemic gaps in the current operating model — not isolated incidents.

🕐
Delayed incident triageEach production incident requires a human to read the alert, search for the relevant runbook, assess severity, and decide who to page. This takes 20–40 minutes per incident. During nights and weekends, incidents can sit untriaged for hours while customer impact accumulates. A P0 raised at 02:18 may not be seen until standup at 09:00.
🔗
Manual and unreliable deploy correlationDetermining whether a recent deployment caused an incident requires an engineer to manually cross-reference incident timestamps against deployment logs. This process is error-prone, time-consuming, and often skipped under pressure. Root causes that would be obvious in hindsight are missed in the moment.
📄
Silent compliance driftVendor contracts are reviewed at signing and then filed. Policy requirements evolve, contracts renew with altered terms, and new vendors are onboarded — but no systematic re-audit occurs. Violations accumulate silently. The current audit identified 9 active violations across two vendors, including a supplier with no termination clause for 5 years and breach notification terms of "in due course."
👁
No continuous operational visibilityThe operations team has no single pane of glass showing incident trends, deployment risk, and compliance status together. Situational awareness requires querying multiple systems. Patterns — such as a service that is repeatedly involved in incidents — are only visible in retrospect during post-incident reviews.
📈
Operational load scales with system complexityAs the number of services, deployments, vendors, and customers grows, the operational burden on the team grows proportionally. The current model does not scale. Adding more engineers to the on-call rotation is not a sustainable answer — the problem requires a structural solution.
Section 03
What Is Agentic AI?

Traditional software executes explicit, pre-defined instructions. Agentic AI is different: it is software powered by a large language model (LLM) that can reason, plan multi-step workflows, read and interpret unstructured data, make judgement calls, and use tools — all from a high-level goal expressed in natural language.

Rather than coding every decision branch ("if error rate > X% AND service = Y, then page team Z"), you describe the goal and the rules, and the agent reasons about how to achieve it. This makes agents uniquely suited to operational tasks, where the data is messy, the rules are nuanced, and the edge cases are endless.

Key properties relevant to this system

PropertyWhat it means in practice
Reads unstructured dataThe agent reads incident bodies, stack traces, runbooks, and contract prose — not just structured fields. It understands that "Timeout waiting for connection from pool" is a connection pool exhaustion symptom without being explicitly told.
Multi-step reasoningThe agent doesn't just match keywords — it reasons: "This incident opened at 14:18. A deploy to the same service went out at 14:04. The deploy touched PaymentService.java, which appears in the stack trace. This is likely the cause."
Tool useThe agent can call external tools: reading files, running shell commands (git pull, gh issue create), querying APIs. It decides which tools to use and in what order based on its reasoning.
Protocol-driven behaviourThe agent's behaviour is governed by a protocol document (CLAUDE.md) — a natural-language specification of what to do, how to decide, and what to output. Changing the protocol is as simple as editing a text file.
Scheduled autonomyThe agent runs on a schedule without human initiation. It is triggered by a cron job (Plan 1) or a repository push event (Plan 2), completes its work, and produces output — all unattended.
On model choice: This system uses Anthropic's Claude — a frontier LLM with strong reasoning, instruction-following, and tool-use capabilities. The agent protocol (CLAUDE.md) is model-agnostic in principle, but the quality of triage reasoning, correlation detection, and compliance analysis is directly dependent on the capability of the underlying model.
Section 04
Proposed Solution

The solution is a system of specialised AI agents that continuously monitor a structured repository of operational data — incidents, deployments, runbooks, and vendor contracts — and produce structured, actionable findings as GitHub Issues, visible in a live operations dashboard.

The repository is the single source of truth. The agents read from it, reason about it, and write their findings back to GitHub. No external databases, no custom infrastructure, no proprietary monitoring tools. The system runs on top of tools the team already uses.

◆
Every incident triaged automatically — severity assigned (P0–P3), matching runbook identified, specific fix steps included, on-call team named. Available within 4 hours on Plan 1; under 60 seconds on Plan 2.
◆
Deploy correlation done for you — every incident is cross-referenced against the deployment log. Causal chains (multiple deploys together causing an issue) are detected. Rollback recommendations are explicit and versioned.
◆
Continuous compliance audit — vendor contracts are re-checked against policy on every run. Violations are filed the moment they are detected, not the next time someone remembers to look.
◆
Live operational dashboard — a single view of all findings: open issues by severity, category trends over day/week/month, a rolling 60-minute live activity feed. Click any data point to drill down to individual findings.
◆
Scales without adding headcount — as the number of services, incidents, and vendors grows, the agent handles it. The operational load on the team does not increase proportionally.
Section 05
Agent Overview

The system is composed of six specialised agents. Each agent has a single, bounded responsibility. They share no direct communication — they interact through the shared repository and GitHub Issues. This design means each agent can be developed, tested, replaced, or paused independently without affecting the others.

1
Plan 2 · Event-Driven

Watcher

Detects new data pushed to the repository and immediately dispatches the Triage and Correlation agents. Replaces the 4-hour polling cycle with sub-60-second event-driven response.

2
Plan 1 Live · Plan 2 Specialised

Triage Agent

Reads each production incident, assigns P0–P3 severity, matches against the runbook library, recommends specific remediation steps, and identifies the correct on-call team.

3
Plan 1 Live · Plan 2 Specialised

Correlation Agent

Cross-references each incident against the deployment log. Identifies which deploy (or combination of deploys) caused each incident. Recommends rollbacks where available.

4
Plan 1 Live · Plan 2 Daily

Compliance Agent

Audits every vendor contract against the 7-rule compliance policy. Files a separate GitHub Issue for each violation with the exact contract clause, the policy requirement, and the specific gap.

5
Active · Dev/Staging Only

Fabricator Agent

Generates realistic synthetic production incidents every 20 minutes from 6 incident templates. Keeps the live dashboard feed active and provides a continuous stream of test data for the triage and correlation agents.

6
Plan 2 · Weekly

Keeper Agent

Audits the agent ecosystem itself. Checks that all scheduled agents are running, runbooks are current, compliance policy has been reviewed, and the GitHub Issues deduplication list is not overflowing.

Section 07
Implementation Approach

The system is being delivered in phases. Phase 1 is live today. Phases 2 and 3 represent the target architecture, with Phase 2 being the priority for the next sprint.

1
Phase 1 — Foundation (Live) Single monolithic Claude Code agent running on a 4-hour durable cron. Covers all three domains (triage, correlation, compliance) in one run. Protocol versioned in CLAUDE.md. 26 GitHub Issues filed. Live dashboard with 6 charts and 30-second refresh. Issue fabricator running every 20 minutes. Global permission allowlist for zero-prompt autonomous operation.
2
Phase 2 — Specialisation (Next Sprint) Migrate from Claude Code cron to GitHub Actions. Split the monolith into 6 specialised agents — one per responsibility. Watcher agent responds to push events in under 60 seconds. Compliance agent runs daily. Keeper agent provides weekly governance. Agents run in GitHub's cloud infrastructure with no local machine dependency and no TTL expiry.
3
Phase 3 — Full Autonomy (Future) P0 escalation via Slack and PagerDuty webhook. CI/CD pipeline automatically appends to the deployment log on every deploy — closing the last manual data gap. Auto-close stale issues. PR-based review gate for protocol changes. Keeper Agent extended with automated remediation suggestions.
Section 08
Design Principles

The following principles guided the architectural decisions in this system. They are documented here so that future extensions and modifications remain consistent with the original intent.

PrincipleWhat it means for this system
Repository as single source of truthAll data the agents read lives in the git repository — incidents, deployments, runbooks, contracts, and the agent protocol. No external databases. Every input is versioned, auditable, and human-readable.
Protocol in version controlAgent behaviour is defined in CLAUDE.md, committed to the repository. Changing what the agent does requires a pull request — creating a natural review gate for behavioural changes.
One agent, one jobEach agent has a single, bounded responsibility. It reads the minimum data it needs and writes exactly one type of output. This makes agents independently testable, replaceable, and debuggable.
GitHub Issues as the message busAgents do not call each other directly. They communicate through the shared state of GitHub Issues — one agent's output can trigger another agent's attention, but there is no direct coupling.
Zero infrastructure preferenceThe system runs on tools the team already has: git, GitHub, Claude Code. No new databases, no custom backends, no managed queues. The dashboard is a single HTML file.
Deduplication by designEvery agent checks for existing findings before creating new ones. Re-runs only produce net-new findings. This makes the system safe to run frequently without creating noise.
Section 09
Assumptions & Constraints
Current constraint — Phase 1: The Claude Code durable cron has a 7-day TTL and requires manual renewal. It also requires a local machine running Claude Code. Phase 2 (GitHub Actions) eliminates both constraints.
ItemDetail
Data quality dependencyThe agent's analysis is only as good as the data in the repository. Incidents missing an opened_at timestamp, or deployments not recorded in recent.json, will be missed or incorrectly analysed.
Runbook coverageThe agent can only recommend runbook-based actions for incidents that match an existing runbook. Novel incident types are flagged but not remediated. The runbook library must be actively maintained.
Deploy data is manual (Phase 1)deploys/recent.json is currently hand-maintained. Phase 2 adds a CI/CD step that appends automatically, but until then, any deploy not recorded cannot be correlated.
GitHub API rate limitsThe dashboard polls the GitHub Issues API unauthenticated (60 req/hour). At 30-second refresh, this exhausts the limit in 30 minutes. Adding an auth token raises the limit to 5,000 req/hour. Required before production use.
Model qualityThe quality of triage, correlation, and compliance analysis depends on the reasoning capability of the underlying LLM. Findings should be treated as informed recommendations, not authoritative determinations.
