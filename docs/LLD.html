Section 02
Deepened Problem Discovery

The HLD identified five high-level problems. This section documents what was discovered behind each problem during technical discovery — the root causes, the evidence from the live repository, and the specific gaps that drove the design of each agent.

2.1 — Triage Latency Is Structural, Not Operational

The operations team reported that P0 incidents were taking 15–40 minutes to receive a severity classification. Investigation revealed this was not caused by slow human response — it was caused by the absence of a structured triage gate. Production incidents arrived as unstructured JSON in a git repository. There was no automated step between "issue file committed" and "engineer reads it." The on-call engineer was expected to simultaneously notice the alert, read the raw JSON, assess severity, and identify the right team — all without tooling.

When incidents arrive at high volume (3–5 simultaneous events is common during deployments), human triage serialises. The fifth incident waited until the first four were assessed. A dedicated Triage Agent eliminates this bottleneck: all incidents in a batch are assessed in parallel within the same run.

2.2 — Deployment Correlation Was Manual and Undocumented

The team maintained a deploys/recent.json file with deployment records. Linking an incident to a causal deployment required an engineer to manually inspect the file, compare timestamps, and form a hypothesis. This process was neither repeatable nor auditable: different engineers reached different conclusions for the same data, and the reasoning was never written down.

Discovery also revealed a causal chain problem: some incidents were caused not by a single deployment but by the interaction of two deployments — for example, a feature flag enabled in one deploy that only triggered a failure because of a schema change in a prior deploy. Manual inspection almost never detected these chains.

2.3 — Compliance Was Point-in-Time, Not Continuous

Vendor contracts were reviewed by legal at signing and rarely afterwards. The compliance-policy.md file — containing the organisation's 7 mandatory clauses — existed but was not operationally connected to the contracts. There was no process that would catch a contract that violated policy as soon as the contract was added to the repository.

Discovery found 9 violations across two of the three active vendor contracts that had never been flagged: Acme Data Platform (3 violations) and Sirius Storage (6 violations). These included CRITICAL-severity breaches of data residency and breach notification requirements.

2.4 — Observability Was Pull-Based and Fragmented

Status information existed across GitHub Issues, the JSON files in the repo, and individual Slack messages. There was no single surface that answered: "What has happened in the last hour?" Engineers checked multiple tabs. Incident commanders refreshed GitHub Issues manually. There was no live chart, no category breakdown, no trend view.

2.5 — The 4-Hour Response Gap

The initial ops agent runs on a 4-hour cron cycle. This means a P0 incident committed to the repository at 12:01 will not be triaged until 16:17 at the latest. For a P0 — active outage with revenue impact — this gap is operationally unacceptable. This gap is a known limitation of the Plan 1 architecture and is the primary driver for the Plan 2 GitHub Actions migration documented in Section 16.

Section 03
Agentic AI & MCP — Technical Depth

The HLD introduced agentic AI at a conceptual level. This section goes deeper — explaining the mechanisms that make autonomous operation possible, the specific role of the Model Context Protocol (MCP) in Plan 2, and the implications of these choices for system design.

3.1 — How an LLM Agent Actually Works

A large language model (LLM) is, at its core, a function that takes text in and returns text out. What makes it an agent is the addition of a tool-use loop: instead of returning a final answer immediately, the model can call external tools (read a file, run a bash command, create a GitHub issue), receive the result, incorporate it into its reasoning, and decide what to do next. This loop continues until the model determines it has completed its task.

Agent Reasoning Loop — Simplified
01 →
Read prompt + context. The model receives the CLAUDE.md protocol, its task, and any initial data.
02 →
Decide: tool or answer? If the model needs information it doesn't have, it calls a tool (e.g., gh issue list).
03 →
Execute tool. The host (Claude Code) executes the tool and returns the result to the model's context window.
04 →
Reason over result. The model reads the tool output and updates its internal state (what it knows, what it still needs).
05 →
Repeat or finish. If more information is needed, loop to step 2. If the task is complete, produce final output.

In this system, the protocol file (CLAUDE.md) is the agent's instruction set. It defines the task boundaries, the decision criteria, the output format, and the deduplication rules. The model reads the protocol at the start of every run and follows it. This means the agent's behaviour can be updated by editing a text file — no code deployment required.

3.2 — Why Protocol-Driven Design Matters

Traditional automation requires explicit, exhaustive rules: if status == "open" and label contains "P0" then page #payments-oncall. These rules are brittle — they break when the data format changes and cannot handle novel scenarios they were not written to handle.

An LLM-based agent applies reasoning to the data, not pattern matching. When the Triage Agent reads a new incident type it has never seen before, it does not fail — it reasons from first principles (what is the customer impact? what service is affected? is revenue at risk?) to arrive at a severity assessment. This makes the system robust to the inevitable variation in real-world incident data.

Practical implication: This means agents are more capable than a rule engine but also require careful protocol design. A vague protocol produces vague output. The precision of CLAUDE.md directly determines the consistency and auditability of the agent's decisions. See Section 11 for the full protocol specification.

3.3 — Durable Cron — The Scheduling Mechanism

Claude Code exposes a CronCreate primitive with a durable: true flag. When set, the scheduled job is written to ~/.claude/scheduled_tasks.json and survives Claude Code restarts. The cron fires a natural-language prompt on the configured schedule. The prompt instructs the agent to pull the repo and follow the protocol — keeping the scheduling layer thin and the intelligence in the versioned CLAUDE.md file.

7-day TTL: Durable cron jobs auto-expire after 7 days. The CLAUDE.md protocol includes re-creation instructions so any operator can restart the schedule without needing to understand the underlying mechanism.

3.4 — Model Context Protocol (MCP) — Plan 2 Relevance

MCP (Model Context Protocol) is an open standard that defines how an LLM agent connects to external data sources and tools at runtime. Rather than hard-coding tool calls into the agent's prompt, MCP allows tools to be registered as servers that the agent can discover and call dynamically.

In Plan 1 (the current implementation), agents use Claude Code's built-in tool set: bash commands, file reads, and pre-authorised gh CLI calls. This is sufficient for the current scope but requires the agent to be invoked locally with an open Claude Code session.

In Plan 2 (GitHub Actions), agents run headlessly inside CI containers. MCP becomes the mechanism by which these containerised agents access:

  • GitHub's API — reading issues, creating issues, querying workflow runs
  • Alerting systems — PagerDuty, Slack webhooks for escalation
  • Internal runbook stores — Confluence, Notion, or internal wikis accessed as MCP servers
  • Monitoring platforms — Datadog, Grafana for enriching triage with live metrics

MCP removes the need to pre-wire every integration into the agent's environment. As the organisation's tooling evolves, new MCP servers are registered — the agent gains new capabilities without its core reasoning protocol changing.

3.5 — Context Window Management

Every LLM has a context window limit — the maximum amount of text it can reason over in a single call. This system is designed to work within this constraint:

AgentContext StrategyWhy
TriageOne issue + three runbook files per callEach issue is independent; small context = fast and cheap
CorrelationOne issue + deploys/recent.json per callCorrelation is pairwise; doesn't need all incidents at once
ComplianceOne contract + compliance-policy.md per callEach contract is audited independently; avoids cross-contamination
KeeperWorkflow metadata + summary counts onlyGovernance is statistical; doesn't need full issue bodies

In Plan 1, the monolithic agent processes all issues in a single run, which means the context window grows with the number of incidents. Plan 2's specialised agents each maintain a small, bounded context — making them faster, more predictable, and cheaper to run.

Section 05
Agent 1 — Triage Agent
Agent 01 / 06
Triage Agent
First responder — assigns severity, links runbooks, identifies the right team

Why This Agent Exists

Without automated triage, every incident requires a human to perform the same structured analysis: read the issue body, assess customer and revenue impact, find the relevant runbook, identify the on-call team. This analysis is repeatable and rule-driven — the exact conditions under which an LLM agent performs reliably.

The Triage Agent exists to eliminate the latency between an incident being raised and the first structured assessment being available. For a P0 event, this is the difference between an engineer reading raw JSON and an engineer reading a triage report that has already identified the runbook, the severity, and the team to page.

Runbook: auth-502runbooks/auth-502-windows.mdFix procedure for auth service 502s during maintenance windows

Runbook: paymentrunbooks/payment-service-degraded.mdFix procedure for payment service NPE and degradation events
CLAUDE.mdRepo rootThe full triage protocol: severity rubric, output format, dedup rules

The Reasoning Chain

The agent does not simply classify by keyword. It reasons through the incident in sequence:

Triage Reasoning Chain — Per Issue
01 →
Parse the incident. Extract: affected service, number and type of customers impacted, error type, duration, whether it is ongoing.
02 →
Assess revenue impact. Is this a payment flow, a checkout path, or a billing integration? If so, treat as potentially P0 regardless of customer count.
03 →
Apply severity rubric. P0: active outage + revenue or multi-enterprise. P1: single enterprise, significant degradation. P2: intermittent, workaround available. P3: no urgency.
04 →
Match runbook. Scan keywords from the issue body against runbook trigger conditions. If match: include runbook link and extract recommended fix steps.
05 →
Assign escalation team. Payment/checkout → #payments-oncall. Auth/login → #platform-oncall. Upload/CDN → #infra-oncall. Product → #product.
06 →
Deduplication check. Query GitHub Issues for existing [Agent Triage] PROD-XXXX title. If found: skip. If not: create.

Section 06
Agent 2 — Correlation Agent
Agent 02 / 06
Correlation Agent
Root cause investigator — links incidents to deployments and surfaces causal chains

Why This Agent Exists

The most common root cause of production incidents is a recent deployment. Identifying this link is the first step in deciding whether to roll back. Without automation, this investigation happens differently every time: different engineers apply different time windows, different keyword matching heuristics, and different standards for what constitutes "enough evidence." The result is inconsistent decisions and undocumented reasoning.

The Correlation Agent makes this process deterministic and auditable. It applies the same 48-hour window, the same service-area matching logic, and the same causal chain detection to every incident — and it writes its reasoning to a GitHub Issue so that the decision is documented regardless of which engineer handles the incident.

Critically, it detects causal chains — situations where no single deployment caused the incident but a combination of two or more did. This is the class of incident that manual correlation almost always misses.

What It Reads

InputLocationKey Fields
Issue JSONissues/PROD-XXXX.jsonopened_at, body, labels (service area keywords)
Deploymentsdeploys/recent.jsonservice, version, deployed_at, deployed_by, commit, summary, rollback_available, last_known_good
CLAUDE.mdRepo rootCorrelation protocol, evidence criteria, output format

The Reasoning Chain

Correlation Reasoning Chain — Per Issue
01 →
Extract issue timestamp. Parse opened_at as ISO 8601. Compute the 48-hour lookback window: opened_at - 48h to opened_at.
02 →
Filter deployments in window. From deploys/recent.json, collect all records where deployed_at falls within the 48h window.
03 →
Match by service area. Payment/checkout/NPE → payment-service. Auth/login/502 → auth-service. Upload/CDN → signing-service. Tenant-specific → tenant-config-service.
04 →
Assess evidence quality. Strong: same service, timing within 30 minutes of incident onset, commit touches relevant code path. Weak: same service but >12h gap, unrelated commit summary. Skip weak correlations.
05 →
Detect causal chains. If two or more deploys are in window for the same service area, analyse whether their interaction could have triggered the issue (e.g., feature flag + schema change).
06 →
Rollback decision. If rollback_available: true, explicitly recommend rollback to last_known_good. State the version clearly.
07 →
Dedup + create issue. Check for existing [Agent Correlation] PROD-XXXX. If new: create with full evidence narrative.

Causal Chain Detection — How It Works

Causal chains are the hardest class of production incident to diagnose manually. A single suspicious deploy is easy to identify; two deploys that each appear safe in isolation but combine to cause a failure require a different kind of reasoning.

Example causal chain (real, from this repo): PROD-4487 (multi-tenant data bleed) was caused by two deploys interacting: tenant-config-service v3.2.1 changed how tenant isolation was enforced, and payment-service v4.8.2 changed how tenant context was passed to downstream calls. Neither deploy alone caused data bleed — together they created a gap in the isolation boundary. The Correlation Agent identified this chain; a manual review of either deploy in isolation would not have.

Output — GitHub Issue Format

Title: [Agent Correlation] PROD-4487 linked to tenant-config-service v3.2.1
Labels: ops-agent, agent-correlation
Body:
  Issue: PROD-4487 — Multi-tenant data bleed, 3 enterprise customers affected

  Correlated Deploy:
    service: tenant-config-service
    version: v3.2.1 — deployed 2026-06-03T14:22Z by deploy-bot
    commit: a7f3c91 — "Refactor tenant isolation middleware"

  Evidence: Deploy occurred 18 minutes before incident onset.
             Commit modifies tenant boundary enforcement in the same
             code path that PROD-4487 identifies as the data leak vector.

  Causal Chain: payment-service v4.8.2 (deployed same day) changed how
                tenant_id is propagated to downstream calls. Combined with
                the isolation refactor, this created a context gap.

  Rollback: RECOMMENDED — rollback to tenant-config-service v3.2.0

  Source: issues/PROD-4487.json, deploys/recent.json
Section 07
Agent 3 — Compliance Agent
Agent 03 / 06
Compliance Agent
Continuous contract auditor — reads vendor contracts against policy and files violations automatically

Why This Agent Exists

Vendor contracts are legal documents written in natural language. Checking them against a structured compliance policy requires reading comprehension, not just pattern matching. Traditional rule-based compliance tools require every policy check to be expressed as a parseable rule against a structured data format — this is only possible if contracts are themselves structured, which they are not.

The Compliance Agent uses the LLM's natural language understanding to read contract prose and apply policy logic to it. It finds violations that a regex or keyword search would miss — for example, a contract that says "audits will be arranged at mutually convenient times" technically satisfies a keyword match for "audit" but violates the policy requirement for ≤30 days' notice on demand.

Discovery found 9 violations across active contracts that had been present and undetected for months. This agent makes compliance continuous: any contract added to the repository is automatically audited on the next daily run.

What It Reads

InputLocationPurpose
Vendor contractscontracts/*.mdFull contract text in markdown; one file per vendor
Compliance policycompliance-policy.mdThe organisation's 7 mandatory clauses with precise requirements
CLAUDE.mdRepo rootAudit protocol, severity mapping, output format

The 7-Rule Policy — With Audit Logic

Each rule requires a different style of reasoning. The agent applies these checks sequentially to every contract:

RulePolicy RequirementSeverityCommon Failure Mode
1. Data ResidencyEU data must remain in EU. No processing outside EU without explicit consent.CRITICALContract is silent on data location; or allows "global infrastructure"
2. Audit RightsAudits on ≤30 days' notice, at least annuallyHIGHContract requires "reasonable notice" or "mutual agreement" — no fixed window
3. TerminationConvenience termination with ≤90 days' noticeHIGHContract requires cause, or notice period exceeds 90 days
4. Liability Cap≥12 months of fees; data breach not excludedHIGHCap is lower than 12 months, or breach scenarios are explicitly excluded
5. Subprocessors≥30 days' written notice before engaging new subprocessorMEDIUMNotice is after the fact, or no notice mechanism stated
6. Breach NotificationNotify within 72 hours of confirmed breachCRITICAL"Promptly" or "as soon as practicable" used instead of a specific window
7. Governing LawMature data protection jurisdiction (E&W, Ireland, EU, US Delaware, or equivalent)MEDIUMGoverning law is an offshore jurisdiction or a state with weak data protection

The Reasoning Chain

Compliance Reasoning Chain — Per Contract
01 →
Read full contract. Process the contract text as context. Identify the vendor, effective date, and contract type.
02 →
Locate relevant clauses. For each of the 7 policy rules, find the clause (or absence of clause) that bears on that rule.
03 →
Apply policy logic. Compare the contract clause to the policy requirement. The comparison is semantic, not lexical — "mutually convenient times" fails the 30-day-notice test even though it contains no disqualifying keyword.
04 →
Quote the violating text. For each violation, extract the exact contract clause that causes the violation. This is the evidence basis for any remediation or negotiation with the vendor.
05 →
Assign severity. CRITICAL for data residency and breach notification. HIGH for audit rights, termination, liability cap. MEDIUM for subprocessors and governing law.
06 →
Dedup + create one issue per violation. Each rule violation is a separate GitHub Issue — granular enough to track remediation individually.

Output — GitHub Issue Format

Title: [Agent Compliance] Sirius Storage: Breach notification window missing
Labels: ops-agent, agent-compliance
Body:
  Contract: contracts/sirius-storage.md — Sirius Storage Ltd
  Rule Violated: Rule 6 — Breach Notification (72-hour window)
  Severity: CRITICAL

  Contract Text:
    "Sirius will notify Customer of any confirmed Security Incident
     affecting Customer Data as soon as is reasonably practicable."

  Policy Requirement:
    Vendor must notify within 72 hours of confirming a breach.
    Vague language ("reasonably practicable") does not satisfy this requirement.

  Gap: No specific time window stated. "Reasonably practicable" is legally
        ambiguous and may not satisfy regulatory notification obligations
        (GDPR Article 33, UK DPA 2018 equivalent).

  Source: contracts/sirius-storage.md, compliance-policy.md

Trigger & Frequency

The Compliance Agent runs daily at 06:00 UTC. Contracts change slowly — a daily cadence is sufficient to catch new contracts added by procurement before they go live. If a contract is added and passes all 7 checks, no issue is created; the agent is silent on compliant contracts.

Section 08
Agent 4 — Watcher Agent
Agent 04 / 06 — Plan 2
Watcher Agent
Event router — detects new incidents on push and dispatches Triage and Correlation agents within 60 seconds

Why This Agent Exists

The 4-hour cron cycle is the primary limitation of the Plan 1 architecture. A P0 incident committed at 12:01 waits until the next cron fire at 16:17. The Watcher Agent eliminates this gap by switching the system from poll-based to event-driven architecture.

The Watcher is the entry point for the Plan 2 agent pipeline. It does not perform analysis itself — its job is to notice that a new incident has arrived and immediately dispatch the agents that should respond to it. This separation of concerns means the routing logic is separate from the analysis logic, and either can be changed independently.

This is a direct application of the event-driven agentic pattern: agents are triggered by state changes in their environment, not by a clock. The git push of a new issue file is the state change that initiates the entire response chain.

Mechanism

The Watcher is implemented as a GitHub Actions workflow triggered by on: push with a path filter on issues/**. When a new JSON file is pushed to the issues/ directory, the workflow:

  • Identifies which files were added or modified in the push
  • Filters for new PROD-*.json files only (ignores modifications to existing issues)
  • Dispatches the Triage Agent workflow with the new issue ID as input
  • Dispatches the Correlation Agent workflow with the same issue ID
MCP relevance: In an MCP-enabled deployment, the Watcher can additionally enrich the dispatch by pulling live metrics from a monitoring platform (Datadog, Grafana) via an MCP server and including them in the context passed to the Triage Agent. This means the Triage Agent receives not just the raw incident JSON but also current error rates, latency percentiles, and affected service metrics — without requiring any change to the Triage Agent's core protocol.

GitHub Actions Workflow

name: agent-watcher
on:
  push:
    paths:
      - 'issues/PROD-*.json'
jobs:
  dispatch:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Identify new issue files
        id: detect
        run: |
          NEW=$(git diff --name-only --diff-filter=A HEAD^ HEAD -- 'issues/*.json')
          echo "new_files=$NEW" >> $GITHUB_OUTPUT
      - name: Dispatch Triage Agent
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.actions.createWorkflowDispatch({
              owner: context.repo.owner, repo: context.repo.repo,
              workflow_id: 'agent-triage.yml',
              ref: 'main', inputs: { issue_files: '${{ steps.detect.outputs.new_files }}' }
            })
Section 09
Agent 5 — Fabricator Agent
Agent 05 / 06
Fabricator Agent
Synthetic incident generator — keeps the demo live and validates the full pipeline end-to-end continuously

Why This Agent Exists

An always-on ops system that is never triggered is indistinguishable from a broken one. The Fabricator solves two problems simultaneously: it provides a continuous stream of synthetic incidents that keep the dashboard and agent pipeline active during demonstrations, and it acts as an end-to-end integration test — every fabricated incident must be triaged and correlated by the ops agents, proving the pipeline is functional.

The Fabricator is explicitly a dev and staging tool only. It is disabled in production via a repo environment variable (FABRICATION_ENABLED=false). All fabricated issues carry the agent-fabricated label to distinguish them from real incidents.

Issue Templates

The Fabricator generates incidents from 6 realistic templates, each modelling a common class of production event. Variable values are randomised within realistic ranges on each run:

TemplateVariablesService Area
Memory pressureUsage: 82–96%. Heap spike 2–4× baselineAny service (12 options)
Intermittent 503sError rate: 8–35%. Duration: 5–25 minAPI gateway / load balancer
DB replication lagLag: 45–300 seconds. Queries affectedDatabase / data platform
TLS cert expiringDays remaining: 3–14Infra / security
Scheduled job missedLast run: 2–48 hours agoBatch processing / ETL
Credential rotation 401sError spike: 200–800% above baselineAuth / API

PROD-ID Generation

The Fabricator does not use a random ID. It scans all existing files in issues/, finds the highest numeric suffix, and increments it by a random value between 1 and 8. This produces a realistic ID sequence (gaps between IDs are normal in real incident management systems) while guaranteeing uniqueness and monotonic ordering.

# Find highest existing PROD-ID
existing = glob.glob('issues/PROD-*.json')
ids = [int(re.search(r'PROD-(\d+)', f).group(1)) for f in existing]
next_id = max(ids) + random.randint(1, 8)
filename = f"issues/PROD-{next_id}.json"

Git Commit & Push

The Fabricator does not just write a file — it commits and pushes to the remote repository. This is essential in Plan 2: the git push is the event that triggers the Watcher Agent and starts the full response pipeline. The commit message format is consistent and filterable:

[fabricator] Add PROD-4571: DB replication lag on analytics-service (auto-generated)

Schedule

In Plan 1, the Fabricator runs every 20 minutes via a separate durable Claude Code cron. In Plan 2, it runs as a GitHub Actions scheduled workflow. The 20-minute cadence ensures the dashboard always has recent activity and exercises the ops agent pipeline multiple times per hour.

Section 10
Agent 6 — Keeper Agent
Agent 06 / 06 — Plan 2
Keeper Agent
Governance guardian — audits the ops system itself weekly to ensure it continues to function as designed

Why This Agent Exists

Automated systems degrade silently. A cron job expires and nobody notices. A runbook becomes 6 months out of date and the ops agent continues referencing it. A compliance policy is not reviewed for a year. The ops issue list grows to 800 items and the dedup query begins timing out. None of these failures raise an error — they just quietly make the system less effective.

The Keeper Agent is the system's self-monitoring layer. It runs once a week and inspects the health of the ops system itself — not the production environment the other agents monitor, but the monitoring infrastructure. This is the meta-layer that prevents the autonomous system from silently degrading into a broken system that appears to run.

What It Checks

CheckThresholdAction if Failed
Scheduled workflow healthNo missed executions in last 7 daysFile governance issue: workflow name + last run timestamp
Runbook stalenessNo runbook older than 90 days without a git commitFile issue: runbook name + days since last update
Compliance policy reviewcompliance-policy.md must have a commit within 180 daysFile issue: policy may need legal review
Issue list sizeOpen ops-agent issues <500File issue: dedup query performance degradation risk
Fabricator statusFABRICATION_ENABLED=false in production environmentFile CRITICAL issue: fabricator is polluting production data

Why Weekly

Governance state changes slowly. A runbook does not become stale in a day. A workflow does not miss three executions without someone noticing. Weekly cadence is frequent enough to catch drift before it causes an operational problem but infrequent enough not to generate noise. The Keeper is scheduled for Monday 08:00 UTC — the start of the working week, when its findings will be read by operations leads in the weekly stand-up.

Agentic Design Note

The Keeper Agent is an example of a meta-agent pattern: an agent whose domain is the behaviour of other agents rather than the external environment those agents monitor. This pattern is directly enabled by the agentic architecture — because all agent outputs (GitHub Issues) and all agent configurations (workflow files, protocol files) are accessible as text, a meta-agent can reason over them using the same tool-use loop as any other agent. No special introspection API is required.

Section 11
The CLAUDE.md Protocol

CLAUDE.md is the versioned instruction set for the ops agent. It is automatically loaded by Claude Code when it enters the repository directory. Every agent protocol — triage rubric, correlation rules, compliance policy checks, dedup logic, output format — is defined here in natural language.

Why Natural Language Protocol?

Traditional automation encodes logic in code. Changing a severity threshold requires a code change, a pull request, a review, and a deployment. CLAUDE.md encodes logic in language. Changing the severity threshold for P1 from "one enterprise customer" to "one or more paying customers" is a one-line edit to a markdown file — no deployment required, and the change is in version control with full git history and attribution.

Protocol versioning: CLAUDE.md is committed to git. This means every change to agent behaviour is auditable: who changed it, when, and what the diff was. In a regulated environment, this git history satisfies audit trail requirements for "how does the automated system make decisions?"

Section 12
Deduplication Engine

Deduplication is not optional — it is the mechanism that makes re-running agents safe. Without it, every cron run would create duplicate issues for every previously-seen incident. The dedup engine is a single query plus a title-matching rule applied before every GitHub Issue creation.

The Dedup Query

gh issue list \
  --repo ruscoe888/always-on-agent \
  --label ops-agent \
  --state all \
  --limit 200

This returns all issues — open and closed — that carry the ops-agent label, up to 200. The agent parses the title list from this output and applies the following matching rules:

AgentDedup KeyExample
Triage[Agent Triage] PROD-XXXX present in titleSkip if any issue title contains [Agent Triage] PROD-4521
Correlation[Agent Correlation] PROD-XXXX present in titleSkip if any issue title contains [Agent Correlation] PROD-4521
ComplianceContract name and rule keyword both present in titleSkip if title contains both Sirius Storage and Breach notification

Why Title-Matching, Not a Database

The dedup engine deliberately avoids a shared state database. A database would require provisioning, access control, and maintenance — adding operational complexity to a system designed for simplicity. Title-matching against the GitHub Issues API is sufficient for the current scale (hundreds of issues) and is naturally consistent: the ground truth is always the live issue list, never a separate store that could drift.

Scale limit: The --limit 200 flag means the dedup query returns at most 200 issues. If the ops-agent issue list exceeds 200 open issues, the dedup check becomes incomplete. The Keeper Agent monitors for this condition (open issue count >500 is a governance alert). At scale, the Compliance Agent's dedup could be made more precise by querying per-contract rather than scanning the full list.

Section 14
Permissions & Security

Autonomous operation requires pre-authorised access to the commands the agent needs. The permission model uses an allowlist — explicit enumeration of permitted commands — rather than broad elevated privileges.

Allowlist — Current (Plan 1)

Stored in ~/.claude/settings.json globally and mirrored in .claude/settings.json at project level. Every command the ops agent needs is listed explicitly — the agent cannot execute anything not on this list without prompting the operator for approval.

"allow": [
  "Bash(git pull*)",        // pull latest repo state before each run
  "Bash(git -C * pull*)",   // pull with explicit path (used by cron prompt)
  "Bash(gh issue list*)",   // dedup query before every issue creation
  "Bash(gh issue create*)", // create triage / correlation / compliance issues
  "Bash(gh label list*)",   // verify labels exist before creating issues
  "Bash(gh label create*)", // create labels if missing
  "Bash(gh api*)",          // direct API calls for advanced queries
  "Bash(cat *)",            // read issue JSON, contract files, runbooks
  "Bash(ls *)",             // list issues/ and contracts/ directories
  "Bash(find *)",           // discover files across the repo
  "Bash(echo *)"            // print run summary
]

The Fabricator is the only agent with write access to the repo and is environment-gated.

Section 16
Plan 2 — GitHub Actions Migration

Plan 1 (Claude Code durable cron) is operational and sufficient for demonstration and initial validation. Plan 2 is the target production architecture: specialised agents running as GitHub Actions workflows, event-driven rather than poll-based, with <60 second response latency for new incidents.

Why Migrate?

⏱
4-hour response gap A P0 incident waits up to 4 hours for its first triage report. This is operationally unacceptable and the primary driver for migration.
🗎
Local dependency Plan 1 requires Claude Code to be running on a local machine. Plan 2 agents run on GitHub's infrastructure — no local machine required, survivable across laptop restarts and network interruptions.
⚖
Monolithic context window The Plan 1 agent processes all incidents in one run, growing the context window with each incident. Plan 2 agents each process one incident with a bounded, predictable context size.

Migration Phases
