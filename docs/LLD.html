Section 01
Data Schemas

All data the agent reads is structured JSON or Markdown. No external databases. Schema integrity is not enforced — the agent reads what it finds and reasons about gaps.

issues/PROD-XXXX.json

FIELD
TYPE
NOTES
id
string
Format: PROD-NNNN. Used as dedup key in issue title matching. Must be unique.
title
string
Human-readable summary. Used for runbook keyword matching and display.
status
string
Always "open" in fabricated data. Agent does not read this field today.
severity
null
Always null on ingestion. The agent assigns severity and writes it to the GitHub Issue, not back to the JSON.
labels
string[]
Source labels from the original system. Not the same as GitHub Issue labels.
assignee
null
Always null. The agent recommends escalation in the GitHub Issue body, doesn't set assignee.
opened_at
string
Critical: Must be ISO 8601 UTC (e.g. 2026-05-19T14:18:23Z). Used as the anchor for the 48h correlation window. Malformed = silent correlation failure.
reporter
string
Email address. Displayed in GitHub Issue body as attribution.
body
string
Full incident description. Agent reads this for keyword matching and context. Should include stack traces, error rates, affected customers.
comments
array
Always empty in fabricated data. Reserved for future use.

{

  "severity": null,

  "reporter": "monitoring-alerts@bts-synthetic.example",

deploys/recent.json

FIELD

Matched against issue keywords for service-area correlation. Must be consistent naming.

Included in correlation issue body for rollback reference.

Currently manual: deploys/recent.json is hand-maintained. Plan 2 adds a CI/CD post-deploy step that appends a new entry automatically, making deploy-to-correlation fully automatic.

contracts/<vendor>.md structure

Vendor contracts are Markdown files with 8 numbered sections (Services, Data Processing, Audit Rights, Term/Termination, Limitation of Liability, Subprocessors, Security Incidents, Governing Law). The agent reads the full text and applies 7 policy rules from compliance-policy.md against each section. No fixed schema — the agent reasons from prose.

Section 02
Cron & Permissions Layer

Cron Schedule

CronJobTTL
17 */4 * * *Ops Agent — triage, correlation, compliance7 days
*/20 * * * *Fabricator — push synthetic issue7 days

Crons are persisted in .claude/scheduled_tasks.json. They survive Claude Code restarts but auto-expire after 7 days. The :17 and :00 offsets avoid fleet contention at round-number marks.

Permissions Allowlist

// ~/.claude/settings.json (global)
{
  "permissions": {
    "allow": [
      "Bash(git pull*)",
      "Bash(git -C * pull*)",
      "Bash(gh issue list*)",
      "Bash(gh issue create*)",
      "Bash(gh label list*)",
      "Bash(gh label create*)",
      "Bash(gh api*)",
      "Bash(cat *)",
      "Bash(ls *)",
      "Bash(find *)",
      "Bash(echo *)"
    ]
  }
}
Permission gap: Any Bash command not in this allowlist will trigger an interactive prompt, which hangs in unattended cron runs. When adding new agent capabilities (e.g., curl for webhooks, python3 for the fabricator), add the command to the allowlist first.

Section 04
Deduplication Logic

Before creating any GitHub Issue, the agent queries all existing ops-agent issues and title-matches against pending findings. This is a substring match, not an exact match.

Dedup Query

gh issue list \
  --repo ruscoe888/always-on-agent \
  --label ops-agent \
  --state all \
  --limit 200
Scale limit: --limit 200 is the current ceiling. If more than 200 ops-agent issues exist (open or closed), the oldest will not be included in the dedup check and duplicates may be created. Mitigation: close resolved issues promptly, or implement pagination.

Match Rules

TaskMatch KeyMatch Rule
TriagePROD-XXXXTitle contains PROD-ID AND [Agent Triage]
CorrelationPROD-XXXXTitle contains PROD-ID AND [Agent Correlation]
ComplianceContract name + ruleTitle contains vendor name AND rule keyword

On match: skip and print SKIPPED: <title> (already reported). On no match: proceed to gh issue create.

Section 06
Fabricator — Component Detail

The fabricator (scripts/fabricate-issue.py) generates synthetic but realistic production issues. It runs every 20 minutes and keeps the live dashboard feed active. It should be disabled in production via the FABRICATION_ENABLED variable.

Template Library (6 types)

#TemplateLabels
1Memory pressure — OOMKill risk (82-96% usage)infrastructure, performance
2Intermittent 503s — error rate 8-35%customer-impact, api
3DB replication lag — 45-300s behind primarydata, infrastructure
4TLS certificate expiring — 3-14 dayssecurity, infrastructure
5Scheduled job missed — 2-48h of executionsdata, reliability
6Credential rotation — 401 spike 200-800% above baselinesecurity, api, customer-impact

PROD-ID Generation

def next_prod_id():
    existing = []
    for f in os.listdir(ISSUES_DIR):
        if f.startswith("PROD-"):
            existing.append(
                int(f.replace("PROD-","")
                     .replace(".json",""))
            )
    base = max(existing) if existing else 4500
    # Random 1-8 gap — organic feel
    return base + random.randint(1, 8)

The random gap (1–8) makes PROD-IDs look organic rather than sequential. IDs are read from disk each invocation — no shared state needed.

Fabricator Execution Flow

1
Select template randomlyPicks one of 6 templates. Selects a random service from 12 options and a random reporter from 5 emails.
2
Generate variablesCalls the template's vars() lambda. Produces random but plausible values (memory %, error rates, customer names, latencies).
3
Find next PROD-IDScans issues/ directory for highest existing ID. Adds a random offset of 1–8.
4
Write JSON fileWrites issues/PROD-XXXX.json with opened_at set to current UTC time in ISO 8601 format.
5
git add → commit → pushCommits with message [fabricator] Add PROD-XXXX: auto-generated issue. Pushes to origin/main.

Live Activity Chart — Bucket Logic

function buildLiveBuckets(issues) {
  const now     = new Date();
  const buckets = [];
  // 60 one-minute buckets, oldest first
  for (let m = LIVE_MINUTES - 1; m >= 0; m--) {
    const start = new Date(now - (m+1) * 60000);
    const end   = new Date(now - m * 60000);
    buckets.push(
      issues.filter(i => {
        const t = new Date(i.created_at);
        return t >= start && t < end;
      }).length
    );
  }
  return buckets;
}

The chart re-ticks every 60 seconds even when no new data arrives (via setInterval(tickLive, 60000)) — so the rightmost bucket always represents "right now" and old buckets scroll left.

Render Strategy

On first load: full DOM is built. On subsequent 30s refreshes: only stat number elements are updated in-place (document.getElementById('val-open').textContent = open.length). Charts are updated via chart.update('none') (no animation on refresh). This prevents layout flash on every poll cycle.
